Sustainability Analytics Dashboard

Role:
Data analyst & developer
Duration:
6 weeks
Stack:
Python, SQL, Power BI, Pandas
Power BI ETL SQL Data Visualization
Problem Statement

Environmental data scattered across incompatible sources

Organizations tracking sustainability metrics (carbon emissions, energy consumption, waste output, water usage) typically collect data in spreadsheets, CSV exports from sensors, and manual logs. These sources use different units, time granularity, and naming conventions. Stakeholder reporting requires consolidating this into coherent, auditable dashboards.

The goal was to build an automated pipeline that ingests data from multiple sources, normalizes it, and feeds a Power BI dashboard that non-technical stakeholders can use to track progress against sustainability targets.

Why This Is Hard

Messy data, inconsistent schemas, and non-technical users

The incoming data had no consistent schema. Energy data was in kWh with hourly readings. Waste data was in kilograms with weekly totals. Carbon data was derived from fuel consumption logs with mixed units (liters, gallons). Joining these datasets requires careful unit conversion and temporal alignment.

Missing data was common — sensors go offline, manual logs get skipped. The ETL pipeline needs to handle gaps without silently producing incorrect aggregations. The difference between "no data recorded" and "zero emissions" matters for audit purposes.

Approach & Key Decisions

Staging-layer ETL with explicit null handling

I designed the pipeline in three stages: raw ingestion (preserve source data as-is), staging (clean, normalize, flag issues), and production (aggregated, dashboard-ready). The staging layer is the critical component — it applies unit conversions, resamples time series to a common daily granularity, and flags rows with missing or suspicious values.

For null handling, I chose to propagate nulls through aggregation rather than impute. A weekly total with 2 missing days shows as "incomplete" in the dashboard, not as a misleadingly low number. This decision trades visual cleanliness for data integrity, which is the right trade-off for sustainability reporting where numbers may be audited.

The Power BI dashboard uses a drill-down structure: top-level KPIs → category breakdown → facility-level detail. Filters are synced across pages so selecting a date range updates all views simultaneously. I used DAX measures for year-over-year comparisons and target-vs-actual calculations.

Dashboard drill-down levels

The automated pipeline processes a full data refresh in under 2 minutes. Previously, compiling the same report manually took 1–2 business days due to the time spent on data collection and formatting.

The null-propagation strategy caught 3 instances where missing sensor data would have produced artificially low emission figures — exactly the kind of error that erodes trust in sustainability reporting.

Trade-offs & Limitations

Power BI dependency and limited automation

The dashboard is tied to Power BI, which requires a license and Windows environment. An open-source alternative (Superset, Metabase) would reduce vendor lock-in but lacks some of Power BI's DAX capabilities and polish for non-technical users.

The ETL pipeline runs on a manual trigger, not a schedule. Adding orchestration (Airflow, Prefect) would make it production-ready but was out of scope for this project timeline.

Future Improvements

Toward automated alerting and predictive targets

  • Add anomaly detection on incoming data to automatically flag sensor malfunctions or data quality issues before they reach the dashboard.
  • Implement scheduled ETL runs with Airflow and add data lineage tracking so every dashboard number can be traced back to its source record.
  • Build a forecasting module that projects whether current trends will meet annual sustainability targets, giving stakeholders early warning.
